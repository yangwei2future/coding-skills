# 阶段5：QA 验证（测试工程师角色）

## 目标
基于 PRD 验收标准和技术方案，执行全面的质量验证，产出测试报告并完成交付。

---

## 执行步骤

### 5.1 加载输入

1. 读取 `docs/<功能名>-prd.md`（提取 BDD 验收标准）
2. 读取 `docs/<功能名>-architecture.md`（理解 API 和数据模型，获取项目上下文摘要中的测试命令）
3. 读取 `docs/<功能名>-tasks.md`（了解实现范围）

### 5.2 识别测试命令

从「项目上下文摘要」中获取测试命令。如果未记录，按以下规则自动识别：

| 特征文件 | 测试命令 |
|:---------|:---------|
| pom.xml | `mvn test` |
| build.gradle | `./gradlew test` |
| package.json (有 test script) | `npm test` 或 `yarn test` |
| pytest.ini / conftest.py | `pytest` |
| go.mod | `go test ./...` |
| Cargo.toml | `cargo test` |
| Makefile (有 test target) | `make test` |

如果无法识别，询问用户。

### 5.3 测试用例生成

从 PRD 的 BDD 验收标准中提取测试场景，补充以下维度的用例：

| 测试类型 | 来源 | 说明 |
|:---------|:-----|:-----|
| 功能测试 | PRD BDD 场景 | 直接从 Given/When/Then 转化 |
| 边界测试 | PRD 决策表 | 覆盖决策表的所有条件组合 |
| 异常测试 | PRD 异常场景 | 网络错误、空数据、无权限等 |
| API 测试 | 技术方案 API 列表 | 请求参数校验、响应格式验证 |
| 集成测试 | 技术方案集成点 | 模块间调用链路验证 |

如果项目配置了 `test-case` skill，优先调用该 skill 生成测试用例。

### 5.4 测试执行

运行已识别的测试命令，记录结果。

### 5.5 缺陷修复循环

如果测试失败，进入修复循环：

```
修复循环（最多 3 轮）：
  1. 分析失败的测试输出，定位根本原因
  2. 区分问题类型：
     - 代码缺陷 → 修复代码
     - 测试用例不合理 → 修正测试
     - 环境/配置问题 → 调整配置
  3. 修复后重新运行失败的测试
  4. 如果仍有失败 → 回到步骤 1
  5. 连续 3 轮仍无法修复 → 标记为已知问题，报告给用户
```

### 5.6 代码质量审查

对所有新增/修改的代码进行审查：

**审查清单：**
- [ ] 无硬编码的敏感信息（密码、密钥、Token）
- [ ] 无硬编码的业务配置值（应通过配置中心注入）
- [ ] SQL 查询使用参数化，无注入风险
- [ ] 用户输入有校验和过滤
- [ ] 异常处理完整，无吞异常
- [ ] 日志记录合理，无敏感信息泄露
- [ ] 并发场景有适当的同步控制
- [ ] 资源（连接、流）有正确的关闭/释放

### 5.7 交付报告

生成测试报告并保存到 `docs/<功能名>-qa-report.md`：

```markdown
# <功能名> - QA 验证报告

## 测试概况
- 测试用例总数：X
- 通过：Y
- 失败：Z
- 跳过：W
- 通过率：XX%

## 测试明细
### 功能测试
| 用例 | 结果 | 备注 |
|:-----|:-----|:-----|
| ... | 通过/失败 | ... |

### 已知问题
| 问题 | 严重程度 | 状态 | 说明 |
|:-----|:---------|:-----|:-----|
| ... | 高/中/低 | 已修复/待修复 | ... |

## 代码审查结果
- [审查发现汇总]

## 交付结论
- [ ] 所有 P0 测试通过
- [ ] 无安全漏洞
- [ ] 代码审查通过
- **结论：** 可交付 / 需修复后交付
```

### 5.8 用户验收

用 AskUserQuestion 询问：
- "QA 验证已完成，请查看报告"
  - 选项：
    - "验收通过，流水线完成"
    - "需要补充测试或修复"
    - "暂停流水线"

---

## 阶段出口检查

- [ ] 测试报告已保存到 `docs/` 目录
- [ ] 所有 P0 用例通过
- [ ] 代码审查无高危问题
- [ ] 用户已验收

---

## 流水线完成

所有阶段通过后，输出最终交付摘要：

```
开发流水线完成

产出文件：
  - docs/<功能名>-prd.md（需求文档）
  - docs/<功能名>-architecture.md（技术方案）
  - docs/<功能名>-tasks.md（任务清单）
  - docs/<功能名>-qa-report.md（测试报告）

实现统计：
  - 新增/修改文件：X 个
  - 新增代码行数：约 Y 行
  - 测试用例：Z 个（通过率 XX%）
```

提交所有改动并推送到远程仓库。
