# 阶段1：需求澄清（产品经理角色）

## 目标
将原始需求转化为 AI 友好的 PRD 文档，确保下游架构师 Agent 能直接解析生成系统设计。

---

## 执行步骤

### 1.1 加载需求原文

根据用户提供的来源获取需求：

- **本地文件** → 用 Read 工具读取
- **飞书文档** → 调用 `feishu-requirement` skill 提取内容
- **直接粘贴** → 从用户消息中提取

将原文保存到 `docs/<功能名>-raw-requirement.md`。

### 1.2 需求缺口分析

对照以下维度扫描原始需求，识别缺失或模糊的部分：

| 维度 | 检查项 |
|:-----|:-------|
| 用户角色 | 涉及哪些角色？各角色权限边界？ |
| 核心流程 | 主流程是否完整？分支和异常路径？ |
| 业务规则 | 条件判断是否穷举？边界值？ |
| UI 状态 | 6态是否覆盖（默认/加载/空/异常/成功/边界）？ |
| 非功能性 | 性能、安全、数据脱敏要求？ |

输出缺口清单，按优先级排列。

### 1.3 交互式澄清

用 AskUserQuestion 工具逐轮与用户澄清缺口：

**规则：**
- 每轮最多问 3 个问题，避免信息过载
- 问题必须具体，给出选项或示例
- 用户回答"不确定"的项标记为 `[待定-合理默认值]` 并注明假设
- 最多进行 3 轮澄清，之后对剩余缺口采用行业最佳实践填充

**澄清优先级：**
1. 核心业务流程的分支逻辑
2. 角色权限边界
3. 异常和边界处理
4. 非功能性约束

### 1.4 生成 AI 友好 PRD

调用 `ai-friendly-prd` skill 生成完整 PRD。

确保 PRD 包含：
- [ ] 至少 1 个 Mermaid 流程图
- [ ] 条件规则用决策表呈现
- [ ] 至少 3 个 BDD 验收场景（正向 + 异常 + 边界）
- [ ] 所有主要页面覆盖 6 态
- [ ] 业务实体使用语义绑定格式
- [ ] 各关键章节包含 Agent 上下文注解（💡）

PRD 保存到 `docs/<功能名>-prd.md`。

### 1.5 用户确认

用 AskUserQuestion 询问：
- "PRD 是否满足需求？"
  - 选项：
    - "确认通过，进入架构设计阶段"
    - "需要修改"（附修改意见后重新生成）
    - "暂停流水线"（保存状态后退出）

---

## 阶段出口检查

- [ ] PRD 文件已保存到 `docs/` 目录
- [ ] 无占位文本或 TODO 标记
- [ ] 用户已明确批准
- [ ] 所有关键缺口已解决或标注了合理默认值

通过检查后，加载 `phase-2-architect.md` 进入下一阶段。
