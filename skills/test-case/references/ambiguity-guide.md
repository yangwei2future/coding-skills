# 常见模糊问题检查清单

在分析需求时，需要特别关注以下容易产生理解偏差的描述类型：

1. **数据返回时机模糊**：
   - 如果描述为"xxx功能返回xxx信息"，需要仔细查看完整描述确认是：
     - "运行结束后返回"
     - "运行过程中持续返回"
     - "运行中和后都返回"
   - 例如不要将"运行中返回"理解为"仅运行结束后返回"

2. **功能性质模糊**：
   - 如果描述为"需要做xxx"、"支持xxx"，需要确认是：
     - 新建功能（之前不存在）
     - 统一/整合现有功能（将多个分散功能整合）
     - 优化/增强现有功能（在现有基础上改进）
   - 例如不要将"统一语法校验"理解为"新增语法校验功能"

3. **状态触发时机模糊**：
   - 如果描述为"xxx时会xxx"，需要确认触发条件：
     - "正在做某事时会xxx" - 运行中触发
     - "当前是xxx时会xxx" - 状态触发
     - "完成后会xxx" - 完成后触发

4. **功能名称相似但描述模糊**：
   - 对于功能名称相似（如"复用参数"、"引用参数"）但描述不清晰的
   - 需要结合上下文和原型图（如果有）确认实际功能

5. **版本/阶段描述模糊**：
   - 注意识别某功能属于哪个阶段（一期、二期等）
   - 不要将"计划中"或"待实现"的功能当作已实现功能编写测试用例

6. **条件性约束规则模糊**：
   - 如果描述包含"仅当...时"、"根据...不同而不同"、"仅支持...类型"等条件性描述，需要仔细确认：
     - **数量约束条件**：如"仅可且必须存在1个"，需明确数量范围和校验时机（保存/提交时）
     - **联动字段约束**：如"仅当字段A为X时，字段B可选Y、Z"，需明确字段间的依赖关系
     - **条件性可选值**：根据某个字段的值，另一个字段的可选项会变化
   - 识别需要校验的条件约束，设计正向和负向测试用例
   - 示例：
     - "模板保存、另存为确认时仅可且必须存在 1个CU沙箱组件"
       - 正向：恰好1个 → 校验通过
       - 负向：0个 → 校验失败，提示"必须包含1个CU沙箱组件"
       - 负向：超过1个 → 校验失败，提示"仅可包含1个CU沙箱组件"
     - "仅模板类型为'轨迹采集标注'时，可选择采集-标注-检查、采集-标注；其他类型选择 仅标注、标注-检查"
       - 正向：轨迹采集标注 → 标注流程可选：采集-标注-检查、采集-标注
       - 正向：其他类型 → 标注流程可选：仅标注、标注-检查
       - 负向：轨迹采集标注 → 标注流程不可选：仅标注、标注-检查

7. **上下文信息缺失**：
   - 文档中引用了外部依赖但未提供详细说明，需要识别并补充：
     - **外部系统依赖**："调用XX服务"、"依赖XX组件"，但未提供接口定义或行为说明
     - **前置功能依赖**："在XX基础上"、"优化XX功能"，但未说明XX的当前功能和状态
     - **数据模型依赖**："引用XX表"、"基于XX数据"，但未提供数据结构定义
     - **业务规则依赖**："按照已有规则"、"遵循XX策略"，但未说明具体规则内容
     - **用户角色/权限依赖**："根据用户角色"、"按照权限"，但未提供角色定义和权限矩阵
   - 示例：
     - "在已有标注模板基础上增加轨迹采集功能" - 缺少现有标注模板的功能说明
     - "依赖CU沙箱组件" - 缺少CU沙箱组件的接口定义和行为说明
     - "按照已有权限体系" - 缺少权限体系说明
   - **处理方式**：使用 AskUserQuestion 询问用户是否需要补充上下文文档，用户可选择：
     - 补充文档：用户提供额外的需求文档或技术方案，skill 将使用多文档模式综合分析
     - 使用假设：基于常见做法进行合理假设，在测试用例中标注假设内容
     - 忽略并继续：按现有文档生成，在测试用例文档中标注缺失的上下文信息
