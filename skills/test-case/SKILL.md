# 测试用例生成器

基于需求文档或技术方案生成高质量的功能测试用例。支持为整个文档或指定的功能模块生成测试用例。

## 使用方法

```bash
/test-case [文档名称] [模块或章节]
/test-case [文档名1,文档名2] [模块或章节]
/test-case [文档名1 文档名2] [模块或章节]
```

**参数说明：**
- 第一个参数（可选）：文档名称或关键词，用于在 docs 目录中查找对应的需求文档或技术方案文档
  - **单文档模式**：直接输入文档名称或关键词
  - **多文档模式**：支持两种分隔方式：
    - 逗号分隔：`需求文档,技术方案`
    - 空格分隔：`需求文档 技术方案`（当某个 token 能在 docs 目录找到匹配文件时，视为文档名而非模块名）
- 最后一个参数（可选）：指定文档中的某个模块、功能或章节名称（仅当该参数无法匹配到 docs 目录中的任何文件时才作为模块名）
- 如果不提供参数，将列出所有可用的文档供用户选择

**支持的文档类型：**
- 需求文档：推荐以"需求_"开头的 .md 文件
- 技术方案：推荐以"技术_"或"方案_"开头的 .md 文件
- 其他文档：任何 .md 文件均可使用，工具会根据内容自动识别类型

**使用场景：**

1. **基于需求文档生成测试用例**
   ```bash
   /test-case 用户管理系统
   ```
2. **基于技术方案生成测试用例**
   ```bash
   /test-case 实时作业DAG图方案

   # 仅生成技术方案中某个模块的测试用例
   /test-case 实时作业DAG图 层级展示
   ```

3. **同时参考需求文档和技术方案**（推荐，生成更全面的测试用例）
   ```bash
   # 逗号分隔（原有方式）
   /test-case 数据质量需求,数据质量技术方案

   # 空格分隔（等价写法）
   /test-case 数据质量需求 数据质量技术方案

   # 综合多文档，针对特定模块生成测试用例
   /test-case 数据集需求,数据集方案 新建版本
   /test-case 数据集需求 数据集方案 新建版本
   ```

4. **生成指定模块的测试用例**（推荐，适用于大文档）
   ```bash
   # 只生成业务规则模块的测试用例
   /test-case 数据质量 业务规则

   # 只生成问题清单模块的测试用例
   /test-case 数据质量 问题清单
   ```

5. **交互式选择**
   ```bash
   /test-case
   ```

## 前置条件

**文档位置：**
- **默认位置**：`docs/` 目录（推荐）
- **其他位置**：可以通过完整路径或相对路径指定文档位置
  - 示例：`/test-case ../other-project/需求.md`
  - 示例：`/test-case /absolute/path/to/document.md`

**文档命名：**
- **推荐命名**：使用 `需求_`、`技术_` 或 `方案_` 前缀便于识别
- **灵活命名**：任何 `.md` 文件名都支持，工具会根据内容自动识别文档类型

**文档来源：**
- 使用 `/feishu-requirement` 从飞书提取（自动保存到 `docs/` 目录）
- 手动创建或上传到 `docs/` 目录
- 指定其他目录下的现有文档

如果用户未提供文档，询问用户文档的位置或建议先获取文档。

## 工作流程

### 第一步：定位文档

**支持单文档和多文档模式**

1. **解析文档参数**
   - **逗号分隔检测**：检查所有参数拼合后是否包含逗号
     - 若包含逗号，则按逗号分隔，得到多个文档关键词 → **多文档模式**
   - **空格分隔检测**：若无逗号，逐个检查每个空格分隔的 token
     - 对每个 token 尝试在 `docs/` 目录中模糊匹配文件名
     - 能匹配到文件的 token → 文档关键词
     - 无法匹配到文件的 token → 模块/章节名（保留给第二步使用）
     - 若匹配到 2 个及以上文档 token → **多文档模式**
     - 若只匹配到 1 个文档 token → **单文档模式**，剩余 token 为模块名
   - 若所有 token 都无法匹配文件，则整体作为单文档关键词（兼容含空格的模糊查询）

2. **查找文档文件**
   - **单文档模式**：在 docs 目录查找包含该关键词的文档
   - **多文档模式**：分别查找每个关键词对应的文档
   - 支持模糊匹配（例如用户输入"DAG图"可以匹配"技术_实时作业DAG图方案.md"）
   - 如果用户未提供参数，列出 `docs/` 目录下所有可用的文档：
     - 需求文档（以"需求_"开头的 .md 文件）
     - 技术方案（以"技术_"或"方案_"开头的 .md 文件）

3. **读取文档内容**
   - **单文档模式**：使用 Read 工具读取单个文档内容
   - **多文档模式**：依次读取所有匹配的文档内容
   - 确认文档可访问且内容完整
   - 识别每个文档的类型（需求文档或技术方案）
   - 如果找不到文档，提示用户并建议先上传文档

4. **识别文档类型**
   - 需求文档：文件名包含"需求_"前缀，通常包含用户场景、产品方案、功能详细说明等内容
   - 技术方案：文件名包含"技术_"或"方案_"前缀，通常包含技术架构、接口设计、技术实现等内容

5. **多文档模式特殊处理**
   - 向用户展示找到的所有文档
   - 确认是否是用户想要的文档组合
   - 如果文档不匹配，允许用户重新选择

### 第二步：确定测试范围

**重要：根据第二个参数（$1）确定生成测试用例的范围**

#### 场景A：用户指定了模块/章节（$1 不为空）

1. **识别目标章节**
   - 用户通过 $1 参数指定了文档中的某个模块或章节（例如："业务规则"、"问题清单"、"层级展示"、"数据接口"）
   - 在文档中查找匹配的章节内容
   - 支持模糊匹配章节标题（如用户输入"DAG图"可匹配"### 实时作业DAG图"、"## DAG图设计"等）

2. **提取目标内容**
   - 提取指定章节的完整内容（包括所有子章节）
   - 保留章节的层级结构
   - 如果找不到匹配的章节，列出文档的主要章节供用户选择

3. **使用 AskUserQuestion 确认范围**
   - 向用户展示找到的章节信息
   - 确认是否是用户想要的测试范围
   - 询问是否需要调整范围

#### 场景B：用户未指定模块（$1 为空）

1. **分析文档结构**
   - 解析文档的章节结构
   - 识别主要功能模块
   - 统计文档大小和复杂度
   - 区分需求文档和技术方案的不同结构特点

2. **智能推荐策略**
   - 如果文档较小（< 10KB），建议生成全量测试用例
   - 如果文档较大（> 10KB），列出主要模块，建议分模块生成
   - 技术方案通常比需求文档更技术化，需关注接口、错误处理、技术特性等

3. **使用 AskUserQuestion 询问用户**
   - 列出文档中的主要模块/章节
   - 询问用户是生成全量测试用例，还是选择特定模块
   - 如果用户选择特定模块，记录用户的选择

### 第三步：理解文档内容

**完整阅读文档，建立上下文**：

**图片内容分析（重要）**：

读取文档后，**必须主动扫描并分析其中的图片**，不得忽略：

1. **扫描图片引用**：在文档内容中查找所有 `![...](...)` 格式的图片引用，提取图片路径
2. **逐一读取图片**：对每张图片尝试加载查看，不能跳过
   - **本地路径**（相对或绝对路径）：直接使用 Read 工具读取
   - **bj.bcebos.com 开头的 URL**：先用 Bash 执行 `curl -s -o <文档所在目录>/<filename> <url>` 下载到文档同目录，再用 Read 工具读取（filename 取 URL 最后一段）
   - **其他远程 URL**：使用 WebFetch 工具；若失败则记录"无法访问"并继续
3. **分类提取信息**：
   - **UI 原型图 / 界面截图**：
     - 识别页面上的所有可操作元素：按钮、输入框、下拉框、开关、列表、表格等
     - 识别各元素的状态：默认态、禁用态、选中态、hover 态、加载态、错误态
     - 识别表单字段及其类型（必填/选填、输入限制、校验规则）
     - 识别页面的操作入口和跳转路径
   - **流程图 / 状态转换图**：
     - 提取完整的节点和流转条件
     - 识别分支判断逻辑及每个分支的路径
     - 找出流程的起始点、终止点和异常出口
   - **数据结构图 / ER 图**：
     - 识别字段约束、枚举值、关联关系
4. **关联上下文**：将图片内容与周围文字说明对照，图片展示的细节（如字段名、状态标签、提示文案）优先于文字的笼统描述
5. **补充测试点**：图片中发现但文字未提及的 UI 细节、状态或约束，需补充到测试用例中

**单文档模式**：
- 理解整体架构和模块关系
- 了解跨模块的依赖关系
- 把握业务流程的上下文
- 理解术语和概念的定义

**多文档模式（重点）**：
- **主次关系原则（核心）**：
  - **第一个文档为主文档**：测试用例的模块结构、功能覆盖范围、测试重点以主文档为准
  - **后续文档为辅助文档**：用于补充主文档中缺失的技术细节、实现方式、边界条件等
  - **生成逻辑**：基于主文档的章节结构生成测试用例框架，从辅助文档中提取补充信息
  
- **综合分析**：需要将多个文档的内容进行关联和融合
  - **以主文档为骨架**：测试用例的模块划分、优先级判断以主文档为基准
  - **用辅助文档补肉**：在主文档的框架下，用辅助文档补充技术实现、异常场景、边界条件
  
- **需求与技术对应**：
  - 需求文档（通常作为主文档）提供业务场景、用户故事、功能说明
  - 技术方案（通常作为辅助文档）提供实现细节、接口定义、数据结构
  - 将需求层面的功能点对应到技术层面的实现方式
  
- **互补信息**：
  - 主文档中可能缺失的技术细节，从辅助文档中补充
  - 辅助文档中可能缺失的业务背景，从主文档中获取
  - 但整体结构和覆盖范围始终以主文档为准
  
- **一致性检查**：
  - 确认主文档和辅助文档描述的是同一功能
  - 识别主文档和辅助文档之间的差异或矛盾
  - 如有不一致，**优先以主文档为准**，除非主文档明显错误或缺失关键信息
  - 若有疑问，使用 AskUserQuestion 询问用户以哪个为准

**特别注意：必须联系全文上下文进行理解和编写**
- 不要仅根据局部描述推断功能行为
- 对于描述不清晰的功能，要结合文档其他部分的相关描述进行理解
- **多文档模式下**，要交叉验证：需求文档中的功能在技术方案中如何实现
- 涉及数据返回、状态变化等场景时，要仔细阅读完整的需求描述和技术实现

**常见模糊问题检查清单**：

遇到数据返回时机、功能性质、状态触发、版本阶段、条件约束、上下文缺失等模糊描述时，参考 [references/ambiguity-guide.md](references/ambiguity-guide.md) 进行判断处理。

**针对目标范围进行深度分析**：

对于要生成测试用例的目标范围（整个文档或指定章节），进行详细分析：
- 识别所有功能点
- 理解业务流程和逻辑
- 提取关键功能点
- 识别输入输出要求
- 注意边界条件和异常场景
- 识别数据约束和验证规则
- 分析与其他模块的交互点
- **重要：检查所有涉及的功能是否已实现（检查版本/阶段说明）**

**状态流转识别（文档中存在状态机时必做）**：

当文档中出现状态流转图、状态枚举、或类似「A → B → C」的描述时，需完整提取以下信息后再生成用例：

1. **绘制状态清单**：列出所有状态名称（含初始态和终止态）
2. **绘制合法转换表**：明确每条合法转换的触发条件和操作者
3. **识别特殊分支**：
   - 分叉（同一状态根据条件走向不同后继状态，如「回归中→已关闭」vs「回归中→重新开放」）
   - 回环（如「重新开放→算法跟进中」使流程重入循环）
4. **识别隐含约束**：
   - 哪些状态是终止态（不可再流转）
   - 哪些转换要求特定角色权限
   - 哪些转换需要前置数据（如必须先填写回归记录才能关闭）

**文档类型特定的分析重点**：

1. **需求文档重点关注：**
   - 用户场景和业务规则
   - 产品原型图（按上述图片分析步骤逐张读取，提取 UI 细节和状态）
   - 前置条件和后置条件
   - 状态转换逻辑

2. **技术方案重点关注：**
   - 接口定义和参数验证
   - 技术特性和边界条件
   - 错误处理和异常场景
   - 性能要求和约束
   - 技术实现细节

3. **多文档模式综合关注：**
   - **模块结构以主文档为准**：测试用例的章节划分、模块命名遵循主文档
   - **测试重点以主文档为准**：哪些功能是核心、哪些是次要，参考主文档的优先级
   - **细节从辅助文档补充**：
     - 主文档中的业务规则如何映射到辅助文档中的技术实现
     - 辅助文档中的接口设计如何支撑主文档的需求功能
     - 数据流转：从主文档描述的数据到辅助文档实现的数据结构
     - 异常处理：主文档中的异常场景在辅助文档中如何处理
     - 性能要求：主文档中的性能指标在辅助文档中如何保障
   - **示例**：
     - 主文档：需求_数据集26H1.md（定义"新建版本"功能）
     - 辅助文档：技术_数据集方案.md（说明版本号生成算法、数据库设计）
     - 生成测试用例时：模块名"新建版本"来自主文档，版本号格式校验细节来自辅助文档

**如有不明确的地方，使用 AskUserQuestion 工具询问用户**：
- 功能范围是否需要调整
- 特定业务/技术规则的细节
- 优先级和重点测试区域
- 与其他模块的交互方式
- **多文档模式下**：需求与技术方案描述不一致时，以哪个为准

### 第四步：生成测试用例文档

基于文档内容，生成全面的测试用例。

**端到端测试用例生成规则**：

在生成测试用例时，分析需求是否涉及**跨系统流转**或**完整业务链路**：

1. **判断是否需要端到端用例**：
   - **需要生成**的场景：
     - 涉及多个系统集成（如：系统A → 中间件 → 系统B → 系统C）
     - 有明确的端到端业务流程描述（如：数据从源头到终点的完整流转）
     - 需求文档中提到"全链路"、"端到端"、"系统间流转"等关键词
     - 涉及数据同步、接口集成、跨系统数据一致性验证
   
   - **无需生成**的场景：
     - 单系统内的功能（如：仅前端UI变更、单个模块功能）
     - 纯API接口测试（无跨系统流转）
     - 纯配置类需求

2. **端到端用例的特征**：
   - **粒度**：端到端
   - **优先级**：通常为P0或P1（验证核心链路）
   - **测试范围**：覆盖从数据产生到最终消费的完整流程
   - **执行步骤**：包含多个系统的操作和验证点
   
3. **端到端用例的命名**：
   - 使用专门的模块：`## 模块X：端到端场景测试`（作为最后一个模块）
   - 用例ID格式：`TC_E2E_NNN`
   - 标题格式：`TC_E2E_001 - 完整流程-<起点>到<终点>`
   
4. **生成策略**：
   - **自动识别**：分析需求文档，如果涉及多系统集成，自动生成1-2个核心端到端用例
   - **用户询问**（可选）：在生成完模块化测试用例后，使用`ask_followup_question`询问用户：
     ```xml
     <ask_followup_question>
     <question>检测到需求涉及跨系统流转（如：研发系统 → 中间件 → Ariba → SAP），是否需要生成端到端测试用例？</question>
     <follow_up>
     <suggest>是，生成端到端用例（覆盖完整业务链路）</suggest>
     <suggest>否，仅保留模块化测试用例</suggest>
     </follow_up>
     </ask_followup_question>
     ```

5. **端到端用例示例结构**：
   ```markdown
   ## 模块六：端到端场景测试
   
   #### TC_E2E_001 - 完整流程-研发系统到SAP
   
   1. **用例信息**
      1. 优先级：P0
      2. 粒度：端到端
      3. 类型：正向
      4. 关联用户故事：作为业务用户，我希望验证数据能在整个系统链路正确流转，以便确保功能完整性
   
   2. **Setup**
      - 所有涉及系统均已完成配置
      - 初始数据已准备
   
   3. **执行步骤**
      1. 在源系统（系统A）创建数据
         - 数据创建成功
      2. 等待中间件同步
         - 中间件表中新增记录
         - 数据推送至下游系统
      3. 在系统B中操作
         - 选择同步过来的数据
         - 提交成功
      4. 验证系统C接收
         - 系统C表中包含正确数据
         - 接口查询返回完整信息
      5. 验证最终消费系统
         - 数据正确展示
         - 业务流程完整
   
   4. **Teardown**
      - 保留测试数据用于后续验证（或根据需要清理）
   ```

6. **注意事项**：
   - 端到端用例不应过多，通常1-3个核心场景即可
   - 重点验证**关键路径**和**数据一致性**
   - 执行步骤应清晰标注**每个系统的操作**和**验证点**

## 测试用例结构

每个测试用例必须严格遵循以下结构（与 common-case-generator 完全一致）：

### 1. 用例信息（必须包含4个有序列表项）
1. **优先级**: P0（核心）/P1（重要）/P2（一般）
2. **粒度**: 单功能/场景/端到端
3. **类型**: 正向/异常/边界/安全/性能
4. **关联用户故事**: 作为{角色}，我希望{目标}，以便{价值}

### 2. Setup（前置条件）
- 使用无序列表（`-`）列出所有前置条件
- 每个条件独立一行
- 条件描述要简洁明确

### 3. 执行步骤（必须使用有序列表+嵌套预期结果）
1. 第一步操作
   - 预期结果1（必须使用`-`开头）
   - 预期结果2
2. 第二步操作
   - 预期结果1
3. 第三步操作
   （无预期结果的过渡步骤）

### 4. Teardown（后置条件）
- 使用无序列表（`-`）列出恢复操作
- 若无恢复操作，写"无"

**格式约束**：
1. 用例标题必须为 `#### TC_模块简称_编号 - 标题`
2. 每个用例必须包含完整的4部分（用例信息/Setup/执行步骤/Teardown）
3. 步骤必须使用 `1.`~`9.` 的有序列表，不能跳过编号
4. 预期结果必须嵌套在步骤下方，用 `-` 开头
5. 步骤之间不能有空行或其他内容

## 测试用例设计原则

1. **完整性**: 覆盖所有功能点和业务场景
2. **正交性**: 每个用例测试一个明确的功能点
3. **可重复性**: 用例可以被重复执行，结果一致
4. **独立性**: 用例之间不互相依赖
5. **可维护性**: 描述清晰，易于理解和更新
6. **可追溯性**: 每个用例都能追溯到具体的需求点或技术要点

### 状态流转专项测试（存在状态机时必须覆盖）

当功能涉及状态流转时，除正常流程外，必须额外生成以下类别的测试用例：

#### 1. 逐步正向流转
- 每一条合法转换路径单独出一条用例，验证触发条件、操作后的状态变更、时间线记录
- 示例：`已上报 → 算法跟进中`、`算法跟进中 → 待回归`、`待回归 → 回归中` 各一条用例
- 分叉路径分别出用例：`回归中 → 已关闭（通过）`、`回归中 → 重新开放（未通过）` 各一条

#### 2. 状态回退（逆向流转）
- 对每个合法转换，验证是否可以反向操作（回退到上一状态）
- 若需求未明确支持回退，则应验证回退操作**被禁止**或按钮不可见/不可点
- 示例：`算法跟进中` 不可回退到 `已上报`；`已关闭` 不可回退到 `待回归`

#### 3. 越级流转（跳跃状态）
- 验证是否可以跳过中间状态直接流转到非相邻的后续状态
- 若需求不支持越级，应验证**越级操作无效**
- 示例：`已上报` 不可直接跳到 `待回归`；`已上报` 不可直接跳到 `已关闭`

#### 4. 终止态验证
- 已到达终止状态后，验证所有流转入口均不可用
- 示例：`已关闭` 状态下推进按钮不显示或置灰

#### 5. 重复触发同一转换
- 对同一转换连续触发两次，验证第二次操作的系统反应（幂等 or 报错）
- 示例：已是「算法跟进中」时再次点击「标记为算法跟进中」

#### 6. 前置条件缺失时的流转
- 某些转换依赖前置数据（如必须填写回归记录才能关闭），验证缺失时流转被阻止并给出提示
- 示例：未提交回归记录时点击「已关闭」流转，应提示必须先新增回归记录

#### 7. 状态与 UI 的一致性
- 每次流转后，验证页面所有与状态相关的 UI 元素同步更新：状态标签、可见按钮、时间线记录、列表行状态

### 边界值与约束条件专项测试（必须覆盖）

当需求中出现任何形式的**数值范围、格式约束、规则限制**时，必须系统性地设计边界值和异常测试用例。

#### 🎯 边界值测试核心方法论

边界值分析（Boundary Value Analysis）是最有效的测试用例设计技术之一。针对任何约束条件，遵循以下策略：

**边界值选取原则** - 对于任何边界值 B，至少测试：
- B-2（明显小于边界）、B-1（紧邻边界）、B（恰好边界）、B+1（紧邻边界）、B+2（明显大于边界）

**组合测试原则** - 当存在多个约束维度时：
- 单维度违反测试
- 多维度同时违反测试
- 边界组合测试

#### 📊 常见边界值测试场景

##### 场景1：数值范围约束

**适用**：年龄、价格、数量、评分、百分比、时长等

**测试策略**：
- **正向**：最小值、中间值、最大值
- **边界**：最小值±1、最大值±1
- **异常**：零值、负数、远超上限、极端值、非法格式

**示例**（年龄18-65岁）：
```
TC_XXX_001 - 输入年龄18岁（最小边界）
TC_XXX_002 - 输入年龄65岁（最大边界）
TC_XXX_003 - 输入年龄17岁时提示年龄过小（边界-1）
TC_XXX_004 - 输入年龄66岁时提示年龄过大（边界+1）
TC_XXX_005 - 输入字母"abc"时提示格式错误（异常）
```

##### 场景2：文本长度约束

**适用**：用户名、备注、描述、标题、评论等

**测试策略**：
- **正向**：最小长度、中等长度、最大长度
- **边界**：最小长度±1、最大长度±1
- **异常**：空字符串、仅空格、特殊字符（emoji、SQL注入、XSS）、极长字符串

**示例**（用户名3-20字符）：
```
TC_XXX_001 - 输入3字符用户名（最小边界）
TC_XXX_002 - 输入20字符用户名（最大边界）
TC_XXX_003 - 输入2字符用户名时提示过短（边界-1）
TC_XXX_004 - 输入21字符用户名时提示过长（边界+1）
TC_XXX_005 - 输入特殊字符（如<script>）时提示格式错误（异常）
```

##### 场景3：日期时间约束

**适用**：预约时间、有效期、生日、起止日期等

**测试策略**：
- **正向**：起始边界日期、中间日期、结束边界日期
- **边界**：当前时刻、前后一天、跨月/跨年边界（2月28/29、12月31）
- **异常**：过去/未来日期（不允许时）、结束早于开始、非法日期（2月30日）、闰年边界

**示例**（预约时间：明天至7天后）：
```
TC_XXX_001 - 预约明天（最小边界）
TC_XXX_002 - 预约7天后（最大边界）
TC_XXX_003 - 预约今天时提示时间过早（边界-1）
TC_XXX_004 - 预约8天后时提示超出范围（边界+1）
TC_XXX_005 - 选择2月29日在非闰年时提示日期无效（异常）
```

##### 场景4：文件/图片上传约束

**适用**：头像、附件、导入文件、图片素材等

**测试策略**：
- **格式**：正向（每种允许格式）、异常（不支持格式）
- **大小**：恰好等于限制、略小/略大限制（±1字节）、远超限制、零字节空文件
- **尺寸**（图片）：符合建议、略偏离建议（±1px）、显著偏离、极端尺寸（1×1、超大）
- **时长**（音视频）：恰好限制、略超限制（±1秒）、远超限制
- **组合**：格式对但大小超、大小对但格式错、多约束同时不满足

**示例**（图片：JPG/PNG，<2MB，建议390×292px）：
```
TC_XXX_001 - 上传符合所有约束的JPG图片（正向）
TC_XXX_002 - 上传恰好2MB的图片（边界）
TC_XXX_003 - 上传2MB+1KB图片时提示超出大小限制（边界+1）
TC_XXX_004 - 上传GIF格式时提示格式不支持（异常）
TC_XXX_005 - 上传尺寸1×1px的极小图片（边界）
TC_XXX_006 - 上传格式对但3MB大小的图片（组合异常）
```

##### 场景5：正则表达式约束

**适用**：手机号、邮箱、身份证号、URL、IP地址等

**测试策略**：
- **正向**：标准格式、边界格式（最短/最长有效）
- **边界**：位数边界、字符边界
- **异常**：缺少必要字符、长度不匹配、格式错误、特殊字符、空格

**示例**（手机号：11位数字，1开头）：
```
TC_XXX_001 - 输入标准11位手机号13812345678（正向）
TC_XXX_002 - 输入10位手机号时提示格式错误（边界-1）
TC_XXX_003 - 输入12位手机号时提示格式错误（边界+1）
TC_XXX_004 - 输入0开头的手机号时提示格式错误（异常）
TC_XXX_005 - 输入包含字母的手机号时提示格式错误（异常）
```

##### 场景6：金额与精度约束

**适用**：价格、余额、转账金额、优惠金额等

**测试策略**：
- **正向**：最小金额、整数金额、最大金额、标准小数
- **边界**：零元、最小单位（0.01）、最大金额±0.01
- **精度**：1位小数、2位小数、3位小数（验证四舍五入）、超长小数
- **异常**：负数、超大金额、非数字、科学计数法
- **业务**：余额不足、优惠后为负、累加精度问题

**示例**（转账：0.01-50000元，2位小数）：
```
TC_XXX_001 - 转账0.01元（最小边界）
TC_XXX_002 - 转账50000元（最大边界）
TC_XXX_003 - 转账50000.01元时提示超出限额（边界+1）
TC_XXX_004 - 转账-100元时提示金额无效（异常）
TC_XXX_005 - 转账123.456元验证小数精度处理（精度）
TC_XXX_006 - 余额100元转账101元提示余额不足（业务边界）
```

##### 场景7：并发与频率限制

**适用**：API限流、验证码、登录尝试、并发操作等

**测试策略**：
- **频率**：刚好在限制内、刚好超出限制、达到/超过限制次数
- **并发**：单请求、接近上限、恰好上限、超过上限、大量并发
- **时间窗口**：窗口内首次、最后一次允许、跨窗口边界、窗口重置后

**示例**（验证码：60秒内只能发送1次）：
```

TC_XXX_002 - 59秒后再次发送提示请稍后（边界-1）
TC_XXX_003 - 60秒后再次发送成功（边界）
TC_XXX_004 - 连续点击5次发送按钮验证限制（并发）
```

##### 场景8：权限与角色边界

**适用**：菜单权限、操作权限、数据权限、功能开关等

**测试策略**：
- **角色**：无权限、最低权限、中等权限、最高权限
- **权限操作**：刚好有权限、刚好无权限、跨权限边界
- **数据范围**：只能访问自己、试图访问他人、访问不存在/已删除
- **状态转换**：权限提升前后、权限降级前后、权限过期

**示例**（普通用户只能查看自己的订单）：
```

TC_XXX_002 - 普通用户尝试查看他人订单时提示无权限（边界）
TC_XXX_003 - 未登录访问订单页面跳转到登录（无权限）
TC_XXX_004 - 用户修改URL参数访问他人数据验证防护（安全边界）
```

##### 场景9：列表与分页边界

**适用**：数据列表、分页查询、批量操作等

**测试策略**：
- **数据量**：0条、1条、恰好1页、跨页、大量数据
- **分页**：第1页、中间页、最后1页、不存在页码（0、负数、超大）
- **每页条数**：最小值、最大值、超出范围
- **批量操作**：选择0条、1条、最大限制、超出限制、跨页选择

**示例**（分页：每页10-100条，批量最多50条）：

### 第五步：保存测试用例文档

1. **生成文件名**
   - 基于需求文档类型和文档名生成对应的测试用例文档名

   - **需求文档 - 部分测试用例格式**：
     - `docs/测试用例_[原文档去除"需求_"前缀后的名称]_[模块名].md`
     - 示例：`docs/需求_数据质量3.0.md` + 模块"业务规则" → `docs/测试用例_数据质量3.0_业务规则.md`
     - 多文档示例：`docs/需求_数据集26H1.md` + `docs/技术_数据集方案.md` + 模块"新建版本" → `docs/测试用例_数据集26H1_新建版本.md`

3. **自动生成XMind并上传飞书思维导图**
   - 保存 Markdown 成功后，**立即自动**执行以下流程：
     1. 读取已保存的测试用例 Markdown 文件，对所有测试用例进行**智能三级归类**，生成思维导图内容：

        **三级层级定义**：
        - **一级节点（`##`）**：与测试用例 .md 文件中的模块划分保持一致，不重新归类
        - **二级节点（`###`）**：在每个一级模块下，将含义相同、功能相近、场景一致的用例归入同一二级节点；不同类型的用例各自保留独立的二级节点，不强制合并；相同类型只保留一个父节点，避免重复拆分
        - **三级内容（`####`）**：具体测试用例，格式 `TC_xxx - 标题`，包含以下子节点：
          - 优先级标签：`##### P0 | 单功能 | 正向`（五级标题）
          - Setup：`##### Setup`（五级标题） + 具体内容（可保持一行，用`;`连接多条）
          - 执行步骤：`##### 执行步骤`（五级标题） + **每个步骤独立展开为六级标题**（`######`）
          - Teardown：`##### Teardown`（五级标题） + 具体内容（可保持一行，用`;`连接多条）

        **归类规则**：
        - 不凭空造内容，所有用例必须来自原文，一条不遗漏
        - 一级节点名称与 .md 中的模块标题保持一致
        - 相同类型/场景的用例合并到同一二级节点下，不同类型各自独立
        - 节点文本超过100字时截断并加 `..`
        
        **执行步骤展开规则（核心）**：
        - 原始Markdown中的执行步骤格式：
          ```markdown
          3. **执行步骤**
             1. 第一步操作
                - 预期结果1
                - 预期结果2
             2. 第二步操作
                - 预期结果3
          ```

          ```markdown
          ###### 1.第一步操作 → 预期结果1 → 预期结果2
          ###### 2.第二步操作 → 预期结果3
          ```
        - **关键**：每个有序列表项（`1.` `2.` `3.`...）必须转换为独立的六级标题（`######`）
        - 预期结果（原`-`列表项）用` → `连接在同一步骤内
        - 步骤编号格式：`数字.内容`（编号后不要空格）

## 测试用例编号规则

- **格式**: TC_[模块简称]_[三位数字]
- **示例**:
  - 用户登录模块：TC_LOGIN_001, TC_LOGIN_002...
  - 订单管理模块：TC_ORDER_001, TC_ORDER_002...
  - 支付模块：TC_PAY_001, TC_PAY_002...
  - DAG图模块：TC_DAG_001, TC_DAG_002...
  - 接口模块：TC_API_001, TC_API_002...

## 优先级划分标准

- **P0**: 核心业务流程，系统基本功能，必须通过
- **P1**: 重要功能，常用场景，高频使用
- **P2**: 一般功能，辅助功能，正常使用场景
- **P3**: 次要功能，边缘场景，低频使用

## 注意事项

1. **详细程度**: 测试步骤要足够详细，让不熟悉系统的人也能执行
2. **数据真实性**: 测试数据要贴近真实业务场景
3. **可执行性**: 确保每个测试用例都是可以实际执行的
4. **维护性**: 使用清晰的命名和结构，便于后续维护
5. **追溯性**: 在测试覆盖矩阵中建立用例与需求点/技术要点的追溯关系
