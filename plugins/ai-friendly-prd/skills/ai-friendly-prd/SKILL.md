# AI-Friendly PRD Generator

生成结构化 PRD，下游 Agent 可直接解析生成系统设计、API 规范和测试用例。

## 执行流程

### Step 1: 需求访谈

读取访谈指南，收集缺失信息：

```
Read: references/interview-guide.md (相对于本 skill 目录)
```

- 先从用户已提供的上下文中提取信息（对话历史、飞书文档、已有需求描述等）
- 只针对明确缺失的部分提问，不要一次问完
- **最小可行信息集**（满足以下条件即可进入 Step 2）：
  - 核心场景：至少 1 个清晰的用户故事或使用场景
  - 业务规则：至少 1 条涉及条件判断的规则
  - 目标用户：明确谁会用这个功能
- 如果用户提供的信息已经很充分，可以跳过访谈直接进入 Step 2

### Step 2: 判断需求复杂度

根据需求特征判断生成完整版还是精简版：

**完整版**（默认）— 需求满足以下任一条件：
- 涉及 2 个以上角色或权限差异
- 包含 2 个以上条件分支的业务规则
- 有 3 个以上页面/屏幕
- 涉及状态流转（如订单状态、审批流程）

**精简版** — 需求满足以下全部条件：
- 单一角色或无角色差异
- 业务逻辑简单（无复杂条件分支）
- 1-2 个页面
- 无状态流转

精简版省略：角色权限模型（第 2 章）、决策表（第 5 章），其余章节保留但可简化。告知用户当前采用哪个版本以及原因。

### Step 3: 生成 PRD

读取 PRD 模板，按模板结构填充真实内容：

```
Read: templates/prd-template.md (相对于本 skill 目录)
```

**生成规则（及原因）：**

1. **流程图必须用 Mermaid 代码块** — 因为下游架构师 Agent 可以直接解析 Mermaid AST 生成状态机代码和 API 调用时序，纯文字描述或图片无法被程序化消费
2. **2 个以上条件的业务规则必须用决策表**（Markdown 表格）— 穷举条件组合可以防止 Agent 生成代码时遗漏分支，也便于 QA Agent 自动生成测试用例矩阵
3. **验收标准必须用 Given/When/Then 格式** — Gherkin 语法是 BDD 框架的标准输入格式，QA Agent 可以直接转化为自动化测试脚本
4. **每个主要页面必须定义 6 种状态**：Default、Loading、Empty、Error、Success、Limit — 前端 Agent 需要这 6 种状态来生成完整的 UI 组件代码，遗漏任何一种都会导致用户看到空白页或未处理的异常
5. **数据绑定用业务实体语义**：`绑定【实体：X】的【属性：Y】`，禁止技术字段名 — 业务语义让 PM 可以校验正确性，同时架构师 Agent 会根据实体定义自动推导 Database Schema 和 API Response 字段
6. **关键章节添加 `> 💡 *给 Agent 的上下文：...`** 注解 — 这些注解替代了人类工程师的"行业常识"和"潜规则"，帮助 Agent 在缺乏领域经验时做出正确决策

### Step 4: 保存与总结

1. 保存到 `docs/[feature-name]-prd.md`
2. 告知用户：
   - 保存路径
   - 采用的版本（完整版/精简版）
   - 包含的 AI 友好元素数量（Mermaid 图 N 个、决策表 N 个、BDD 场景 N 个等）
   - 下一步建议（如：可以用 `/dev-pipeline` 进入开发流水线）

### Step 5: 迭代修订（用户反馈后）

用户对 PRD 初稿提出修改意见时：

1. 定位需要修改的章节，只改动相关部分，不要重写整个文档
2. 修改后重新执行质量检查清单
3. 更新版本号（v1.0 → v1.1）和最后更新日期
4. 告知用户本次修改了哪些内容

## 质量检查清单

生成完成后，逐项自检（精简版可跳过标注 ★ 的项）：

- [ ] 至少 1 个 Mermaid 图
- [ ] ★ 决策表覆盖所有多条件业务规则
- [ ] 至少 3 个 BDD 场景（正向 + 异常 + 边界）
- [ ] 主要页面定义了全部 6 种 UI 状态
- [ ] 业务实体语义绑定（无原始字段名）
- [ ] Agent 上下文注解（至少 3 处 💡 注解）
- [ ] 非功能约束已实质填写（不是占位文本）
- [ ] 无 TODO、TBD 或 [...] 占位文本
- [ ] Mermaid 图语法正确（可在 Mermaid Live Editor 渲染）
