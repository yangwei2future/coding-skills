# 阶段3：任务拆解（技术负责人角色）

## 目标
将技术方案拆解为原子级编码任务，并按依赖关系分组为可并行执行的「波次」。每个任务的描述必须自包含——编码 Agent 只需读取任务描述即可开始工作，不需要再去翻阅完整的技术方案。

---

## 执行步骤

### 3.1 加载输入

1. 读取 `docs/<功能名>-prd.md`
2. 读取 `docs/<功能名>-architecture.md`（重点关注「项目上下文摘要」部分）

### 3.2 原子任务分解

将技术方案拆解为独立的编码任务，每个任务满足：

**原子任务标准：**
- 单一职责：一个任务只做一件事
- 可独立验证：有明确的完成标准
- 预计代码量：50-300 行（超过则继续拆分）
- 依赖明确：清楚列出前置任务

**任务描述模板：**

每个任务必须包含以下信息，因为编码 Agent 会直接拿着这段描述去工作：

```markdown
### 任务 [波次号].[序号]: [任务名称]

- **类型：** 新增 / 修改 / 配置
- **涉及文件：** [预期修改的文件路径，尽量精确到具体文件]
- **依赖任务：** [前置任务编号，无则填"无"]
- **项目约定：**
  - [从项目上下文摘要中提取的相关约定，如分层架构、命名规范、配置方式等]
- **实现要点：**
  - [具体要做什么]
  - [关键逻辑说明]
  - [需要参考的已有代码路径和模式]
  - [需要注意的边界条件]
- **验收标准：**
  - [可验证的完成条件]
- **自验证：**
  - [编码完成后应执行的验证命令，如编译、单元测试]
```

### 3.3 并行分组（波次编排）

按依赖关系将任务分配到执行波次：

```
波次1（并行执行）: 无依赖的基础任务
  ├── 任务1: 数据模型/Entity 定义
  ├── 任务2: 配置项新增
  └── 任务3: 工具类/常量/枚举定义

波次2（并行执行）: 依赖波次1的任务
  ├── 任务4: 数据访问层（DAO/Repository/Mapper）
  ├── 任务5: 第三方服务集成
  └── 任务6: 权限配置

波次3（并行执行）: 依赖波次2的任务
  ├── 任务7: 业务逻辑层（Service）核心
  └── 任务8: 业务逻辑层（Service）辅助

波次4（并行执行）: 依赖波次3的任务
  ├── 任务9: 接口层（Controller / API）
  └── 任务10: 前端页面（如有）

波次5: 集成和收尾
  └── 任务11: 集成联调和端到端验证
```

**分组规则：**
- 同一波次内的任务之间无依赖，可完全并行
- 波次间严格按顺序执行
- 每个波次建议 2-5 个并行任务
- 修改同一文件的任务不能放在同一波次（避免冲突）

### 3.4 生成任务清单

输出完整的任务清单，格式：

```markdown
# <功能名> - 编码任务清单

## 项目上下文（摘自技术方案）
[粘贴项目上下文摘要的关键信息]

## 总览
- 总任务数：N
- 波次数：M
- 预计涉及文件：X 个

## 波次 1：[波次主题]
### 任务 1.1: [名称]
...

## 波次 2：[波次主题]
### 任务 2.1: [名称]
...
```

保存到 `docs/<功能名>-tasks.md`。

### 3.5 用户确认

用 AskUserQuestion 询问：
- "任务拆解是否合理？"
  - 选项：
    - "确认通过，开始编码"
    - "需要调整"（附调整意见后重新拆解）
    - "暂停流水线"

## 阶段出口检查

- [ ] 任务清单已保存到 `docs/` 目录
- [ ] 任务清单开头包含项目上下文摘要
- [ ] 每个任务有明确的文件路径和验收标准
- [ ] 每个任务包含自验证步骤
- [ ] 波次依赖关系合理，无循环依赖
- [ ] 同波次任务无文件冲突
- [ ] 用户已明确批准

通过检查后，加载 `phase-4-coding.md` 进入下一阶段。
