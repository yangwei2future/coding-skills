# 阶段4：并行编码（开发者角色）

## 目标
按波次顺序执行编码任务，同一波次内的任务通过 Agent 工具并行执行。每个编码 Agent 拿到的 prompt 必须自包含所有必要信息，不依赖 Agent 自己去搜索上下文。

---

## 执行步骤

### 4.1 加载输入

1. 读取 `docs/<功能名>-tasks.md`（任务清单，含项目上下文摘要）
2. 读取 `docs/<功能名>-architecture.md`（技术方案，用于提取每个任务相关的设计细节）

### 4.2 创建任务追踪

使用 TaskCreate 为每个编码任务创建追踪项，使用 TaskUpdate 设置波次间的依赖关系（blockedBy）。

### 4.3 波次执行

按波次顺序执行，**每个波次开始前先更新状态文件**：

```json
// 更新 docs/<功能名>-pipeline-state.json
{
  "current_phase": 4,
  "phase_status": "in_progress",
  "coding_progress": {
    "current_wave": 2,
    "total_waves": 4,
    "completed_waves": [1],
    "updated_at": "2024-01-15T11:00:00"
  }
}
```

每个波次的流程：

#### 步骤 A：准备 Agent Prompt

为当前波次的每个任务，从技术方案和任务清单中提取信息，组装完整的 Agent prompt。

**Agent Prompt 模板：**

```
你是一个专注的编码智能体，负责实现一个独立的编码任务。

## 项目上下文
[从技术方案开头的「项目上下文摘要」中粘贴，包含：]
- 技术栈：[语言 + 框架]
- 分层架构：[如 Controller → Service → Mapper → Entity]
- 配置方式：[如 Apollo 配置中心，字段默认值通过配置注入]
- 代码风格：[命名约定、注释风格等]

## 你的任务
[粘贴任务清单中该任务的完整描述，包含：]
- 任务名称和类型
- 涉及文件路径
- 实现要点
- 验收标准

## 技术方案参考
[从架构文档中提取与此任务直接相关的设计片段，如：]
- 相关 API 设计（路径、请求/响应结构）
- 相关数据模型（表结构、字段定义）
- 相关业务规则

## 相关已有代码参考
[如果任务涉及修改或参考已有文件，用 Read 先读取关键片段粘贴在这里，
让 Agent 知道已有代码的风格和模式，而不是让它自己去猜]

## 编码要求
1. 严格遵循项目现有的代码风格和命名规范
2. 配置项的默认值通过配置中心注入，不要硬编码
3. 处理异常和边界条件
4. 不要修改任务范围外的文件
5. 不要添加额外的功能或"改进"

## 完成后
1. 执行自验证：[粘贴任务描述中的自验证命令]
2. 报告：修改了哪些文件、实现了什么功能、是否有需要注意的事项
```

#### 步骤 B：并行启动 Agent

对当前波次的每个任务，用 Agent 工具启动子智能体：

```
Agent:
  prompt: [上面组装好的完整 prompt]
  run_in_background: true
  description: "编码任务 X.Y: [简短描述]"
```

**同一波次内的所有 Agent 调用必须放在同一条消息中并行发送。**

#### 步骤 C：等待波次完成并验证

所有子智能体完成后：

1. **检查 Agent 输出**：逐个检查每个 Agent 的报告，确认无报错
2. **标记完成**：用 TaskUpdate 将完成的任务标记为 `completed`
3. **按文件粒度 git commit**：对本波次产生的每个变更文件，单独提交一个 commit 并立即 push，commit message 遵循项目规范
4. **编译检查**：运行项目构建命令（从项目上下文摘要中获取），确认无编译错误
5. **冲突检查**：如果多个 Agent 修改了相邻文件或同一模块，检查是否有逻辑冲突
6. **依赖检查**：确认下一波次的前置条件已就绪

如果发现问题，在当前波次内修复后再进入下一波次。

> **按文件粒度提交规则**：每个文件单独一个 commit，不合并提交。例如同一波次改了 3 个文件，应执行 3 次 `git add <file> && git commit && git push`，而非一次 `git add .`。

### 4.4 异常处理

| 异常情况 | 处理方式 |
|:---------|:---------|
| Agent 执行失败 | 读取 Agent 输出的错误信息，分析原因后重新启动 Agent（补充缺失的上下文信息） |
| Agent 修改了范围外的文件 | 用 git diff 检查，撤回范围外的修改 |
| 文件冲突 | 手动合并冲突部分，确保逻辑一致 |
| 编译失败 | 定位失败原因，直接修复（不需要重启 Agent） |
| Agent 超时 | 检查任务是否太大，考虑进一步拆分后重试 |

### 4.5 重复波次执行

重复 4.3 的步骤直到所有波次完成。

### 4.6 全量集成验证

所有波次完成后：

1. **构建验证**：运行完整构建命令，确认编译通过
2. **基础测试**：运行测试命令（从项目上下文摘要中获取），确认无回归
3. **代码审查**：快速扫描所有新增/修改的代码，检查：
   - 是否有硬编码值（应通过配置注入）
   - 是否有遗漏的异常处理
   - 是否有不一致的命名
4. **确认提交完整**：验证所有文件变更均已按文件粒度提交并 push，无遗漏

如果集成验证发现问题，直接修复（修复的文件同样需要单独 commit 并 push，不需要重新走任务拆解）。
